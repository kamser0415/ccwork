# tdd-green

TDD의 **GREEN** 단계만 담당한다. 지금 빨간 테스트를 **통과시키는 최소한의 코드**를 쓴다.
테스트를 새로 쓰지 않고(그건 `tdd-red`), 리팩터링도 하지 않는다(그건 다음 REFACTOR 단계).

**왜 "최소"인가?** GREEN의 목적은 "테스트가 요구하는 행동을 가장 단순하게 만족"하는 것이다.
테스트에 없는 기능을 미리 만들면 ①검증되지 않은 코드가 늘고 ②다음 RED가 가짜로 통과(false GREEN)할
위험이 커진다. 테스트가 명세(spec)다 — 테스트가 시키는 것만, 시키는 만큼만 구현한다. 욕심내서 일반화·추상화하는
대신, 빨간 것을 초록으로 바꾸는 가장 짧은 경로를 택한다. 정리·일반화는 모든 테스트가 초록이 된 뒤의 몫이다.

**RED→GREEN은 "본문 채우기"다.** `tdd-red`는 없는 대상 파일에 `throw new Error('not implemented')`
골격을 남겼다. GREEN은 그 throw 본문을 **실제 로직으로 교체**하고, 이미 있던 파일은 테스트가 요구하는
동작을 갖도록 고친다. 파일·시그니처는 유지하고 **본문만** 바뀐다("삭제 후 재생성"이 아니다).

## 입력

- `$ARGUMENTS` = **GitHub 이슈 번호**. 선행 정수만 파싱한다(예: `"2 통과시켜줘"` → `2`).
- 입력 정본: `docs/features/tag/issue-{N}.md`의 **`## 테스트 시나리오`**(통과시킬 행동의 목록)와
  **`## 확정 시그니처`**(구현할 파일·타입). 이 파일이 없으면 멈추고 `test-scenarios`/`tdd-red`부터 안내한다.
- 실제 빨간 테스트가 입력이다. `npm test`로 빨간 게 하나도 없으면, 이미 GREEN이거나 RED 단계가 안 된 것이니
  무엇이 이상한지 보고하고 멈춘다(임의로 새 테스트를 만들지 않는다 — 그건 이 스킬 일이 아니다).

## 산출물

- 빨간 테스트를 통과시키는 **`src/` 구현 코드**(throw 골격의 본문 채우기 + 기존 파일 동작 수정).
- 통과 진행 상황을 반영해 갱신된 `docs/features/tag/issue-{N}.md`(시나리오 항목 체크).
- 마지막 커버리지 측정 결과 요약(미커버 라인 = 빠진 케이스 후보)과 전체 결과 보고.

## 비협상 제약

이 제약들이 GREEN을 GREEN답게 만든다 — 어기면 "테스트 주도"가 깨진다.

- **테스트에 없는 기능은 구현하지 않는다.** 시나리오/테스트에 단언이 없는 동작은 만들지 않는다(YAGNI).
- **테스트 파일은 절대 수정하지 않는다.** 테스트가 틀린 것 같으면 고치지 말고 **개발자에게 보고**한다.
  (테스트를 고쳐 통과시키는 건 명세를 몰래 바꾸는 것 — RED 단계의 신뢰를 무너뜨린다.)
- **불필요한 추상화 금지.** 지금 한 곳에서만 쓰는데 인터페이스·제네릭·헬퍼 레이어를 미리 만들지 않는다.
  중복은 모든 테스트가 초록이 된 뒤 REFACTOR에서 정리한다.
- **디자인 문서에 없는 시각 요소를 임의로 추가하지 않는다.** `docs/design/`의 스펙은 그대로 적용하되,
  스펙에 없는 색·간격·아이콘·애니메이션을 창작하지 않는다(필요해 보이면 보고).
- **시나리오에 있는 행동만, 시그니처 그대로** 구현한다. 시그니처를 바꾸고 싶으면 멈추고 보고한다
  (시그니처 변경은 `test-scenarios` 게이트의 영역).

## 이 프로젝트의 환경 (요약)

- 테스트: Vitest 3 + RTL 16 + user-event 14, `globals: true`/`jsdom`(`vite.config.ts`). `describe`·`it`·`expect`·`vi` 무-import.
- 실행: 단일 파일 `npx vitest run <파일>`(빠른 루프), 전체 `npm test`(= `vitest run`, 회귀 확인), 커버리지 `npx vitest run --coverage`.
- 아키텍처·코드 컨벤션은 **`CLAUDE.md`가 정본**이다(계층: API → Context → 컴포넌트, 네이밍, async 패턴, 검증은 `console.error`·`alert` 금지 등). 구현 전 해당 부분을 따른다.
- Vitest는 esbuild로 타입만 제거하므로 **타입 에러로는 실패하지 않는다.** 실패는 런타임 호출/단언에서 난다.

## 실행 절차

### 1. 실패 목록 확인 — `npm test`

전체를 한 번 돌려 **빨간 테스트 목록**과 각 실패 이유를 확보한다. 이게 이번 GREEN의 작업 목록(work-list)이다.
시나리오 1개 ↔ 테스트 1개이므로, 빨간 테스트 = 아직 구현 안 된 행동이다.

### 2. collect(수집) 실패 감지 → 없는 모듈만 골격 생성

테스트가 **실행조차 안 되고 import 단계에서 깨지는 것**(collect/transform 에러)을 먼저 걸러낸다.
이게 남아 있으면 일부 테스트가 "보이지 않아" GREEN 작업 목록이 불완전해진다.

**결정 신호는 vitest의 collect 에러 메시지다** — 개수 비교는 보조 힌트일 뿐이다. 둘을 혼동하면 오작동한다.

먼저 vitest 출력에서 **collect/import 에러**를 확인한다: `failed to collect` / `Cannot find module` /
`Failed to resolve import` / transform 에러. (`npm test 2>&1 | grep -iE 'failed to collect|Cannot find module|Failed to resolve import'`)

- **collect 에러가 있으면** → import 대상 모듈이 없는 것이다. `tdd-red`의 골격 규칙대로 **확정 시그니처를
  그대로 가진 stub 파일**을 **없는 파일에만** 만들되 본문은 `throw new Error('not implemented')`로 둔다
  (기본값 반환 금지 — false GREEN 방지). 이미 있는 파일은 건드리지 않는다. 만든 뒤 `npm test`를 다시 돌려
  전부 실행(collect 통과)되는지 확인한다. 이 테스트들은 이제 `not implemented`로 빨개지며 §4의 작업 목록에 들어온다.

```ts
// 예: src/lib/tags.ts 가 아예 없을 때만 — 시그니처대로, 본문은 throw (행동 없음)
export function parseTagInput(_input: string): string[] {
  throw new Error('not implemented');
}
```

**보조 힌트 — 시나리오 수 vs 테스트 총수:** `issue-{N}.md`의 시나리오 불릿 수와 vitest의 `Tests` 총수(T)를 비교한다.
(`grep -cE '^\s*- (\[정상\]|\[경계\]|\[예외\]|\[ \]|\[x\])' docs/features/tag/issue-{N}.md` vs `Tests   N failed | M passed (T)`의 T)

- **collect 에러가 있어 T가 모자랐던 거라면** → 위에서 골격을 만들면 T가 늘어 해소된다.
- **collect 에러는 없는데 T < 불릿 수라면** → 모듈 부재가 아니라 **`tdd-red`가 일부 시나리오를 아직 테스트로 안 옮긴 것**이다.
  이 경우 **stub을 만들지도, 테스트를 새로 쓰지도 않는다**(테스트 작성은 `tdd-red`의 영역). 어떤 시나리오가 테스트 없이
  남았는지 **개발자에게 보고**하고, 있는 빨간 테스트로 GREEN을 진행할지 먼저 `tdd-red`로 돌아갈지 확인한다.
- T = 불릿 수이고 collect 에러도 없으면 → 작업 목록이 완전하다. §3으로 간다.

### 3. 디자인 컨텍스트 로드 (UI 구현이 있을 때만)

이번 이슈에 컴포넌트 구현이 포함되면, 코드를 쓰기 전에 시각 스펙을 먼저 파악한다. 우선순위:

1. **`design-system` 스킬 / `docs/design/`** — 이 프로젝트의 디자인 정본. `docs/design/README.md`를 먼저 읽고,
   구현 대상에 맞는 파일만 추가로 읽는다(컴포넌트=`components.md`, 색=`colors.md`, 간격·반경=`spacing-radius.md`,
   타이포=`typography.md`, 그림자=`elevation.md`, 토큰 매핑=`token-map.md`, 금지 패턴=`do-dont.md`).
   → 대상 컴포넌트의 **색상·간격·레이아웃·반경·상태별(기본/hover/focus/disabled/선택) 표현**을 확정한다.
2. **`docs/features/tag/issue-{N}.md`의 "디자인 참고" 섹션** — 있으면 함께 확인(이슈 특화 시각 요구).
3. **`CLAUDE.md`의 스타일링 컨벤션** — 토큰 어휘·가드레일(raw 색/인라인 hex 금지 등). 항상 참조한다.

디자인 문서가 전혀 없으면(다른 프로젝트 등) 이 단계를 건너뛰고 **`CLAUDE.md` 컨벤션만** 따른다.
일반 `docs/design-system.md` 단일 파일을 쓰는 프로젝트라면 그 파일을 읽는다 — 요지는 "구현 대상 컴포넌트의
시각 스펙을 어디서든 찾아 확보한다"는 것이다. 못 찾으면 임의 디자인 대신 그 사실을 보고한다.

### 4. 첫 번째 실패 테스트를 통과시키는 **최소 구현**

빨간 테스트 하나를 고른다(의존 순서상 **아래 계층부터**가 자연스럽다: 순수 함수 → 훅 → 컴포넌트 → Context → 통합).
그 **테스트가 요구하는 행동만** 구현한다:

- throw 골격이면 → throw를 지우고 **테스트를 통과시킬 최소 로직**으로 본문을 채운다.
- 이미 있는 파일이면 → 단언이 깨지는 지점만 고친다(예: `fetchNotes`가 `tags`를 `[]`로 보정하도록).
- **UI 컴포넌트**면 → 테스트가 검증하는 동작(렌더·콜백·접근성 라벨 등)을 충족하되, **§3에서 확정한
  디자인 스펙(Tailwind 클래스·레이아웃 구조·상태별 표현)을 그대로 적용**한다. 동작과 스타일을 한 번에 입힌다.
  단 디자인 문서에 없는 시각 요소는 더하지 않는다(§제약).
- 한 번에 **테스트 하나**를 겨냥한다. 여러 개를 한꺼번에 구현하려다 회귀를 못 잡는 것보다 낫다.

### 5. 통과 확인 + 회귀 확인 — 피드백 루프 (최대 5회)

방금 구현을 검증한다:

1. **대상 파일만** 빠르게: `npx vitest run <대상 테스트 파일>` → 목표 테스트가 초록인지 본다.
2. **전체 회귀**: `npm test` → 다른 테스트가 깨지지 않았는지 확인한다.
3. 목표 테스트가 아직 빨가면: **실패 메시지를 끝까지 읽고 원인을 분석**해 구현을 고친 뒤 5번을 다시 돈다.
   - "기대 vs 실제" 차이, 던져진 에러, 셀렉터 부재 등을 근거로 **최소한만** 수정한다(테스트는 못 고친다).
   - 회귀가 났으면 방금 변경이 원인이다 — 그 변경을 좁히거나 바로잡는다.
4. **같은 테스트를 5회 고쳐도 통과 못 하면**: 멈추고 개발자에게 **실패 메시지·시도한 수정·추정 원인**을 보고한다.
   (테스트나 시그니처 자체가 의심되면 그 점을 명시한다 — 임의로 테스트를 고쳐 통과시키지 않는다.)

### 6. 시나리오 항목 체크

방금 통과한 테스트에 해당하는 `issue-{N}.md`의 시나리오 줄을 **완료 표시**한다.

- 줄이 `- [ ]` 체크박스면 → `- [x]`로 바꾼다.
- 줄이 `- [정상]/[경계]/[예외] ...` 불릿이면(이 프로젝트 형식) → 줄 끝에 ` ✅`를 붙인다(카테고리 prefix는 보존).

이렇게 문서가 곧 진행 현황판이 된다(어디까지 GREEN인지 한눈에).

### 7. 다음 실패 테스트로 → §4 반복

빨간 테스트가 없어질 때까지 §4–6을 반복한다. 매 사이클은 "한 테스트 통과 → 전체 회귀 확인 → 체크"로 닫는다.

### 8. 전체 GREEN 후 커버리지 측정 (참고용)

모든 테스트가 초록이 되면: `npx vitest run --coverage`

- **미커버 라인**을 확인한다. 이건 "시나리오가 놓친 케이스"일 수 있는 신호다.
- 의미 있는 미커버(분기·에러 경로 누락 등)는 **개발자에게 보고**한다 — "이 케이스가 시나리오에 빠진 것 같다"는 식으로.
- **커버리지를 올리려고 테스트를 추가하지 않는다.** 커버리지는 진단 도구일 뿐, 새 테스트는 `test-scenarios`/`tdd-red`의 영역이다.
  (참고: `vite.config.ts`에 `reportOnFailure: true`가 켜져 있어, 혹시 실패가 남아도 리포트는 생성된다.)

### 9. 결과 요약 보고

다음을 요약한다:

- 통과시킨 테스트 수 / 전체(예: `16/16 GREEN`)와, 5회 루프로도 실패해 남긴 항목(있으면).
- 만지거나 새로 만든 `src/` 파일 목록(골격 본문 채움 / 기존 파일 수정 구분).
- `issue-{N}.md`에서 체크한 시나리오.
- 커버리지 수치와 **미커버로 발견한 빠진 케이스 후보**(개발자 판단 요청).
- 다음 단계 제안(REFACTOR로 중복 정리 등은 별도 단계임을 명시).

## 자주 만나는 케이스 레시피

이번 프로젝트(태그 기능)에서 GREEN이 자주 부딪히는 지점들:

- **순수 함수(`parseTagInput`)**: 쉼표 분리 → `trim` → 빈 조각 제거. 길이 초과 등 검증은 spec대로
  **무시 + `console.error`**(throw 아님 — `CLAUDE.md`·spec 근거). 테스트가 요구하는 정규화만 구현한다.
- **훅(`useTagInput`)**: `useState`로 `tags`/`tagInput` 보유, `commit`/`reset` 등 반환. 함수형 업데이트(`setTags((prev) => ...)`).
  중복 제거(dedup)는 해당 이슈(#4)의 테스트가 요구할 때만 — #2에선 중복 허용(테스트가 그렇게 단언).
- **프레젠테이셔널 컴포넌트(`TagInput`/`TagChipList`)**: 동작(콜백·`isComposing` 가드·키 렌더)을 충족하고,
  §3 디자인 스펙(칩 스타일·간격·입력 필드 토큰)을 적용. 한글 IME는 `e.nativeEvent.isComposing` 검사로 Enter 커밋을 막는다.
- **API 계층(`fetchNotes`/`createNote`)**: 타임스탬프 주입 등 기존 패턴 유지하면서 `tags`만 흐르게 한다.
  `fetchNotes`는 옛 노트 호환을 위해 `tags`를 `[]`로 보정(테스트가 단언).
- **Context(`createNote` 액션)**: API와 동사 통일, `import * as api`로 호출, 응답 노트로 `setNotes` 갱신(재요청 없음).
- **통합(`NoteEditor` 저장·재진입·취소)**: api 모듈 모킹 + `NotesProvider` 환경에서 폼↔영속 흐름을 잇는다.
  취소 시 추가한 태그를 폐기하는 동작 등은 테스트 단언을 정확히 만족하는 선에서만 구현.

## 주의

- GREEN은 **빨강을 초록으로**가 전부다. 코드가 못생겼어도 통과하면 일단 멈춘다 — 정리는 REFACTOR.
- 한 번에 테스트 하나. "여러 개를 몰아서 구현 → 전부 빨강"보다 "하나씩 → 매번 전체 회귀 확인"이 빠르고 안전하다.
- 막히면(5회 초과) 혼자 끌지 말고 보고한다. 테스트/시그니처의 모순을 발견하는 것도 가치 있는 산출이다.
- 사용자 노출 텍스트·`alert` 금지·주석은 한국어 등 `CLAUDE.md` 컨벤션을 구현 내내 지킨다.
