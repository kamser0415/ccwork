# tdd-refactor

TDD의 **REFACTOR** 단계만 담당한다. **모든 테스트가 초록인 상태를 한순간도 깨지 않으면서** 코드 구조를 다듬는다.
새 동작을 추가하지 않고(그건 다음 RED), 기능을 바꾸지도 않는다 — **겉보기 동작은 고정, 안쪽 구조만 개선**한다.

**왜 "동작 고정, 구조만"인가?** 리팩토링의 정의가 그렇다 — _관찰 가능한 동작을 바꾸지 않고_ 내부 구조를 개선하는 것.
그래서 **통과 중인 테스트가 안전망**이 된다: 구조를 바꿔도 테스트가 계속 초록이면 동작이 보존됐다는 증거다.
테스트가 빨개지면 그건 "구조 개선"이 아니라 "동작 변경"이라는 신호이므로 **즉시 되돌린다**. 이 피드백 루프가
리팩토링을 안전하게 만든다 — 그래서 한 번에 하나씩 바꾸고 매번 테스트로 확인하는 규율이 핵심이다.

## 입력

- `$ARGUMENTS` = **GitHub 이슈 번호**. 선행 정수만 파싱한다(예: `"2 정리해줘"` → `2`).
- 실제 입력은 **이번 이슈에서 바뀐 `src/` 코드 + 전부 초록인 테스트**다. 통과 안 하는 테스트가 하나라도 있으면
  리팩토링을 시작하지 말고 멈춘다(그건 `tdd-green`이 끝낼 일 — 빨간 상태에선 안전망이 없다).
- 컨벤션 정본은 `CLAUDE.md`, 기능 맥락은 `docs/features/tag/issue-{N}.md`(필요 시 참고).

## 산출물

- 동작은 동일하나 더 깔끔한 **`src/` 코드**(중복↓·이름 명확·책임 분리·복잡도↓·컨벤션 정합).
- **여전히 전부 초록인 테스트**(테스트 파일·시나리오 변경 없음).
- **리팩토링 전후 요약**: 무엇을 왜 바꿨는지, 적용/롤백한 항목, 테스트 그린 유지 확인.

## 비협상 제약

이 제약들이 "리팩토링"을 "몰래 기능 바꾸기"와 구분한다.

- **새 기능·새 동작 추가 금지.** 테스트가 요구하지 않는 동작을 더하지 않는다(그건 RED부터 다시 시작할 일).
- **테스트 파일 수정 금지.** 테스트는 안전망이라 손대면 망의 의미가 사라진다. 그래서 리팩토링은 **테스트가 의존하는
  공개 계약(export 이름·시그니처·동작) 안쪽**에서만 한다. 만약 개선이 그 계약(테스트가 쓰는 이름 등)을 바꿔야 한다면
  멈추고 보고한다 — 테스트를 고쳐 맞추지 않는다.
- **`src/` 밖 수정 금지.** 설정·문서·테스트 외 파일은 건드리지 않는다.
- **대상 파일 범위(아래 §3 식별 결과) 밖 수정 금지.** 이번 이슈가 건드린 파일만 정리한다. 범위 밖 파일을 손대야 하거나
  새 파일을 만들어야 깔끔해진다면(예: 공통 헬퍼 추출) **임의로 하지 말고 §4 게이트에서 함께 보고·승인**받는다.
- **한 번에 하나의 변경**, 매번 `npm test`. 여러 개를 몰아 바꾸면 무엇이 테스트를 깼는지 못 가린다.

## 이 프로젝트의 환경 (요약)

- 테스트: Vitest 3, 전체 `npm test`(= `vitest run`), 단일 파일 `npx vitest run <file>`. esbuild라 타입에러로는 실패 안 함.
- **컨벤션 정본은 `CLAUDE.md`**: 계층(API→Context→컴포넌트), 네이밍(`handleXxx`/`onXxx`, `isXxx`, API↔Context 동사 통일),
  async 패턴(try/catch/finally, 함수형 업데이트), 가드절 정렬, 디자인 토큰(@theme만, raw 색 금지) 등.
- 편집 시 디자인 가드 훅(PostToolUse `scripts/check-design.mjs`)과 커밋 시 `lint-staged`(eslint/prettier/디자인)가 돈다.
  리팩토링도 이 규칙을 깨면 차단되므로 **raw 색·미사용 변수 등을 새로 들이지 않는다**.

## 실행 절차

### 1. 컨벤션 파악 — `CLAUDE.md` 읽기

프로젝트별 네이밍·계층·패턴·스타일링 규칙을 먼저 머리에 넣는다. "정합성" 판단의 기준이 이 문서다.

### 2. 출발점 확인 — `npm test` 전부 초록

```bash
npm test
```

- **하나라도 빨가면 멈춘다.** 빨간 상태에선 안전망이 없어 리팩토링을 시작하면 안 된다(먼저 `tdd-green`).
- 전부 초록임을 눈으로 확인한 이 시점이 **롤백 기준점**이다. → §5의 롤백이 안전하려면 이 기준점이 **커밋되어** 있어야 한다.
  GREEN이 아직 미커밋이면 먼저 커밋(또는 `git stash` 스냅샷)해 깨끗한 기준점을 만든다. 그래야 실패한 변경을
  `git checkout -- <file>`로 되돌려도 GREEN 작업을 잃지 않는다.

### 3. 리팩토링 대상 파일 식별

이번 이슈가 바꾼 **`src/` 프로덕션 파일**만 대상이다. 변경 파일은 **커밋분 + 작업 트리(미커밋)** 를 합쳐 구한다:

```bash
# 커밋된 변경(브랜치 기준) + 미커밋(스테이징/언스테이징/신규)을 합집합으로
{ git diff main...HEAD --name-only -- src/;
  git diff --name-only -- src/;
  git diff --cached --name-only -- src/;
  git ls-files --others --exclude-standard -- src/; } | sort -u
```

- ⚠️ **`git diff main...HEAD`만 쓰지 않는다.** GREEN 변경이 아직 커밋 전이면 그 명령만으론 안 잡힌다(흔한 함정).
  위처럼 작업 트리 변경까지 합쳐야 NoteEditor·Context 등 미커밋 파일이 누락되지 않는다.
- **테스트 파일(`*.test.ts`/`*.test.tsx`) 제외.** 남은 프로덕션 파일이 점검 대상이다.
- 각 대상 파일을 읽고 아래 **점검 기준**으로 개선 후보를 모은다.

#### 점검 기준 (범용 — CLAUDE.md와 함께 본다)

- **중복 제거**: 같은 로직/상수/패턴이 여러 곳에 반복되는가? (한 곳으로 모을 가치가 있는지 — 단, 성급한 추상화는 금물)
- **네이밍 명확성**: 이름이 의도를 드러내는가? 약어·모호·오해 소지 없는가? (CLAUDE.md 네이밍 규칙 정합)
- **단일 책임**: 한 함수/컴포넌트가 너무 많은 일을 하는가? 쪼개면 읽기 쉬워지는가?
- **불필요한 복잡도**: 매직 넘버·과도한 중첩·죽은 코드·**불필요한 추상화**(한 곳서만 쓰는데 인터페이스/제네릭로 감쌈)?
- **컨벤션 불일치**: CLAUDE.md의 계층·패턴·async·가드절·디자인 토큰과 어긋나는가?

> 후보가 없으면 그렇게 보고하고 끝낸다 — **리팩토링은 "할 게 있을 때만" 한다.** 멀쩡한 코드를 굳이 비틀지 않는다.

### 4. ▣ 게이트 — 대상 목록 보고 후 승인 대기

모은 개선 후보를 **파일별로, "무엇을·왜·어떻게(요지)"** 로 정리해 개발자에게 제시하고 묻는다:
"이 항목들을 리팩토링할까요? 뺄/추가할 것 있으면 알려주세요." **승인 전까지 코드를 바꾸지 않는다.**
범위 밖 파일/새 파일이 필요한 항목이 있으면 여기서 함께 밝힌다(임의 진행 금지). 수정 요청은 반영 후 재제시.

### 5. 한 건씩 적용 → `npm test` → 통과 확인 (반복)

승인된 항목을 **하나씩** 처리한다. 한 항목마다:

1. 그 변경 **하나만** 적용한다(여러 개 묶지 않는다).
2. 영향 파일을 빠르게: `npx vitest run <관련 테스트 파일>` → 이어서 전체 `npm test`로 회귀 확인.
3. **전부 초록이면** 그 변경을 채택하고 다음 항목으로.

### 6. 깨지면 즉시 롤백 → 다른 접근

테스트가 하나라도 빨개지면, 방금 변경이 **동작을 바꿨다**는 뜻이다(=리팩토링 실패).

- **즉시 되돌린다**: `git checkout -- <file>`(기준점이 커밋돼 있을 때) 또는 방금 편집을 역으로 취소.
- 되돌려 다시 초록임을 확인한 뒤, **더 작은 단위로 쪼개거나 다른 방법**으로 시도한다. 두세 번 시도해도 안전하게
  못 바꾸면 그 항목은 **보류**하고 이유를 보고한다(억지로 밀어붙여 빨간 채로 두지 않는다).

### 7. 전후 요약 보고

다음을 요약한다:

- **적용한 리팩토링**: 파일별 변경 항목과 "무엇을 왜"(예: "NoteEditor의 저장 핸들러 중복 분기 → 헬퍼로 추출").
- **보류/롤백한 항목**과 이유(테스트가 깨져서 / 계약을 바꿔야 해서 등).
- **테스트 상태**: 시작도 끝도 `npm test` 전부 초록(개수 명시), 테스트 파일 무수정.
- 다음 단계 제안(예: 변경분 커밋, 다음 이슈의 RED).

## 리팩토링 vs 동작 변경 — 헷갈리는 경계

"이건 리팩토링인가, 동작 변경인가?"가 애매하면 **테스트가 답을 준다.** 순수 리팩토링은 테스트를 **수정 없이** 통과시킨다.

- **리팩토링(OK)**: 변수/함수 이름 변경(공개 계약 밖), 중복 합치기, 함수 추출, 가드절 정렬, 매직 넘버 상수화,
  주석/포맷 정리, 불필요한 분기 단순화 — 모두 _입출력·렌더 결과 동일_.
- **동작 변경(이 스킬 아님)**: 새 분기/검증 추가, 기본값 변경, 에러 처리 방식 변경, 공개 시그니처/이름 변경
  (테스트가 의존하면 테스트를 고쳐야 함) — 이런 게 필요하면 멈추고 보고하거나 RED 단계로 넘긴다.

## 주의

- REFACTOR는 "구조를 더 낫게"가 목적이지 "완벽하게"가 아니다. **과한 추상화·범위 확장은 오히려 후퇴**다 — 점검 기준에
  걸리는 것만, 분명히 좋아지는 것만 손댄다.
- 한 번에 하나, 매번 테스트. "여러 개 몰아 바꾸고 한꺼번에 테스트"보다 **원자적 변경 + 즉시 검증**이 안전하고 빠르다.
- 디자인 토큰·ESLint를 새로 어기지 않는다(편집 훅·커밋 훅이 막는다). 한글 텍스트·주석 컨벤션 등 CLAUDE.md를 끝까지 지킨다.
- 끝났을 때 **테스트 개수·통과가 시작과 동일**해야 한다(테스트가 늘거나 줄면 리팩토링 범위를 벗어난 것).
