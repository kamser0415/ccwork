# security-review

TDD 사이클(REFACTOR)까지 끝난 코드를 **커밋하기 직전**에 한 번 거르는 게이트다. 세 가지를 본다:
**타입 오류**(빌드 깨짐), **의존성 취약점**(npm audit), **환경변수(.env) 노출**(비밀이 코드에 새는지).
핵심은 "다 고치기"가 아니라 **정확히 분류해서 진짜 위험한 것만, 승인받고 고치기**다.

**왜 분류가 핵심인가?** 스캐너 결과를 전부 고치려 들면 ①false positive·의도된 패턴까지 건드려 멀쩡한 걸
깨뜨리고 ②dev 전용 취약점처럼 배포에 무관한 것에 시간을 쓴다. 위험의 **맥락**(배포되는 코드인가? 비밀이
실제로 새는가? 빌드를 깨는가?)으로 우선순위를 갈라야, 커밋을 막을 진짜 이슈만 처리하고 나머지는 기록으로
남긴다. 그래서 이 스킬은 "수집 → 3분류 → 승인 → 즉시수정만 → 재스캔" 순서를 지킨다.

## 입력

- `$ARGUMENTS` = **GitHub 이슈 번호**. 선행 정수만 파싱한다(예: `"2 점검해줘"` → `2`).
- 실제 입력은 **커밋 직전의 작업 트리**다. 이번 이슈에서 바뀐 코드가 대상이지만, 스캔 도구(tsc/audit)는
  저장소 전체를 보므로 **이번 이슈와 무관한 기존 항목**이 섞여 나온다 → §분류에서 맥락으로 가른다.

## 산출물

- **3분류 보고**: 수집한 타입오류·취약점·.env 노출을 "즉시 수정 필요 / 권장 수정 / 무시 가능"으로 나눠 근거와 함께 제시.
- 승인 후 **"즉시 수정 필요" 항목만 처리**한 변경(있을 때).
- **재스캔 결과**: 즉시수정 항목이 해소돼 클린인지 확인.

## 비협상 제약

- **판단이 필요한 수정은 반드시 승인 후.** 자동으로 고치지 않는다 — 무엇을 왜 고칠지 보고하고 승인받은 항목만 손댄다.
- **테스트 파일(`*.test.ts`/`*.test.tsx`) 수정 금지.** (tsc가 테스트 파일 에러를 뱉어도 — §환경 참조 — 그건 고칠 대상이 아니다.)
- **`npm audit fix --force` 금지.** `--force`는 메이저 버전을 멋대로 올려 빌드/동작을 깨뜨린다. 취약점 수정은
  `npm audit fix`(비-force) 또는 **대상 패키지만 명시적 버전 조정**으로, 그것도 "즉시 수정 필요"로 승인된 것만.
- **"즉시 수정 필요"만 처리.** 권장·무시 항목은 보고만 하고 건드리지 않는다(범위 폭주 방지).

## 이 저장소의 맥락 (분류에 꼭 반영)

이 프로젝트는 **학습용 노트 앱**(로컬 실행, 프로덕션 배포 아님)이다. 그래서 분류 시 아래를 전제한다:

- **tsc의 테스트 전역 오류는 기존 설정 이슈다.** `tsconfig.json`이 `include:["src"]`라 테스트 파일까지
  타입체크하는데 vitest 전역 타입이 없어, `tsc --noEmit`이 `*.test.*`에서 `Cannot find name 'describe'/'it'/'vi'/'expect'`를
  대량으로 뱉는다. **이건 이번 이슈의 타입 오류가 아니라 기존 설정 미비의 증상**이다 → 프로덕션 `src` 파일 오류와
  반드시 분리한다(아래 §1).
- **dev 의존성 취약점은 대개 배포 무관.** `json-server`·`esbuild`·`vite` 등 개발 도구 체인의 취약점은
  빌드 산출물(클라이언트 번들)에 실리지 않는다. 런타임(`dependencies`)에 실려 배포되는 패키지의 취약점과 구분한다.
- **의도된 로컬 하드코딩.** `src/api/notes.ts`의 `API_URL = 'http://localhost:3001'`은 비밀이 아니라 로컬 개발용 상수다.
- 단, 맥락이 "무시 가능"의 핑계가 되어선 안 된다 — **실제 비밀 유출·런타임 취약점·프로덕션 빌드 깨짐은 맥락과 무관하게 "즉시 수정 필요"**다.

## 실행 절차

### 1. 타입 오류 수집 — `npx tsc --noEmit`

```bash
npx tsc --noEmit 2>&1 | tee /tmp/sec-tsc.txt
# 프로덕션(src, 테스트 제외) 오류만 분리 — 이게 진짜 신호
grep -E 'error TS' /tmp/sec-tsc.txt | grep -vE '\.test\.(ts|tsx)' || echo "→ 프로덕션 타입오류 없음"
# 테스트 파일 전역 오류(기존 설정 이슈)
grep -E 'error TS' /tmp/sec-tsc.txt | grep -E '\.test\.(ts|tsx)' | head
```

- **프로덕션 `src` 파일의 오류** → 빌드(`tsc && vite build`)를 깨므로 **즉시 수정 필요** 후보.
- **테스트 파일의 전역 오류**(describe/it/vi/expect) → 기존 설정 미비(§맥락) → **무시 가능** 또는 "권장: tsconfig에 `types:["vitest/globals"]`"로 분류. 자동 수정 안 함.

### 2. 의존성 취약점 수집 — `npm audit`

```bash
npm audit 2>&1 | tee /tmp/sec-audit.txt | tail -30
npm audit --json > /tmp/sec-audit.json 2>/dev/null   # 패키지별·심각도별 상세
```

- 각 취약점이 **런타임(`dependencies`)인지 dev(`devDependencies`)인지**, **직접 의존인지 전이(transitive)인지**,
  **실제로 이 앱에서 도달 가능한지**를 본다. (`npm ls <패키지>`로 어디서 끌고 오는지 추적 가능.)
- high/critical이라도 dev 전용·배포 무관이면 즉시수정이 아닐 수 있다 — 심각도 라벨만 보고 판단하지 않는다.

### 3. 환경변수(.env) 노출 점검 — grep

```bash
ls -a | grep -E '^\.env' || echo "(.env 파일 없음)"
# src/에서 환경변수 직접 참조
grep -rnE 'process\.env|import\.meta\.env' src/ || echo "(직접 참조 없음)"
# 흔한 비밀 흔적(키/토큰/시크릿이 코드에 박혔는지)
grep -rniE '(api[_-]?key|secret|token|password|private[_-]?key)\s*[:=]' src/ || echo "(하드코딩 비밀 흔적 없음)"
```

- `.env` 파일이 있으면 그 **값**들이 `src/`에 리터럴로 박혔는지(번들에 유출) 확인한다:
  ```bash
  # .env의 값들을 추출해 src/에서 검색(빈 줄·주석 제외)
  while IFS='=' read -r k v; do [ -n "$v" ] && grep -rnF "$v" src/ && echo "  ⚠️ 위 위치에 .env 값($k) 유출"; done < .env 2>/dev/null
  ```
- **Vite 주의**: `VITE_` 접두사 env만 클라이언트 번들에 노출된다(의도된 동작). 비밀은 `VITE_`로 노출하면 안 되고,
  비밀 **값**이 코드에 하드코딩되면 유출이다 → **즉시 수정 필요**.

### 4. 3분류 보고 (수집 결과 전부)

모은 항목을 아래 기준으로 나눠, 항목마다 **무엇을·어디서·왜 이 등급인지** 근거와 함께 제시한다.

- **🔴 즉시 수정 필요** — 보안 위험 또는 빌드 실패 수준:
  - `src/` 프로덕션 코드에 비밀 값 하드코딩 / `.env` 값 유출 / 비-`VITE_` 비밀의 클라이언트 노출.
  - 프로덕션 `src` 파일 타입 오류로 `tsc && vite build`가 깨짐.
  - **배포되는 런타임 의존성**의 high/critical 취약점 중 이 앱에서 도달 가능한 것.
- **🟡 권장 수정** — 코드 품질 저하(커밋은 막지 않음):
  - 빌드는 통과하나 약한 타입(`any` 남용 등), dev 의존성의 moderate 취약점,
    tsconfig에 vitest 전역 타입 추가(테스트 tsc 노이즈 제거) 같은 설정 개선.
- **⚪ 무시 가능** — false positive 또는 의도된 패턴:
  - 테스트 파일의 describe/it/vi 전역 오류(기존 설정 증상), dev 전용·배포 무관 취약점,
    `API_URL='http://localhost:3001'` 같은 의도된 로컬 상수.

### 5. ▣ 게이트 — 승인 후 "즉시 수정 필요"만 처리

3분류 보고를 제시하고 묻는다: "🔴 항목을 이렇게 처리할까요? 등급 조정/제외할 것 있으면 알려주세요."
**승인 전까지 아무것도 고치지 않는다.** 승인된 🔴 항목만, 한 번에 하나씩 처리한다:

- 타입 오류 → 해당 `src` 파일 최소 수정(테스트 파일 금지). 동작을 바꾸면 안 되므로 수정 후 `npm test`로 회귀 확인.
- 취약점 → `npm audit fix`(비-force) 또는 대상 패키지 명시 버전 조정. **`--force` 금지.** 수정 후 `npm test`로 회귀 확인.
- 비밀 유출 → 값을 코드에서 제거하고 `.env`/환경변수로 이전(필요 시 `.gitignore` 확인). 이미 커밋·노출된 비밀이면 **로테이션 필요**를 보고.

### 6. 재스캔 → 클린 확인

처리한 종류만 다시 돌려 🔴가 해소됐는지 확인한다(`npx tsc --noEmit` / `npm audit` / 해당 grep).

- 🔴가 모두 사라지고 `npm test`가 여전히 전부 초록이면 커밋 준비 완료.
- 남은 🟡·⚪는 "이번엔 처리 안 함"으로 명시해 보고한다(은폐 금지).

### 7. 요약 보고

- 3분류 최종 목록, 처리한 🔴 항목과 방법, 재스캔 결과(클린 여부), `npm test` 상태(개수).
- 처리 안 한 🟡·⚪와 그 이유, 후속 제안(예: tsconfig 전역 타입 추가는 별도 chore).

## 주의

- **심각도 라벨 ≠ 우선순위.** critical이라도 dev 전용·도달 불가면 즉시수정이 아닐 수 있고, moderate라도 런타임
  비밀 유출이면 즉시수정이다. 항상 **맥락(배포되나·비밀이 새나·빌드를 깨나)** 으로 판단한다.
- 맥락을 "무시 가능"의 핑계로 남용하지 않는다 — 애매하면 🟡로 올려 보고하고 개발자가 판단하게 한다.
- 이 스킬은 **점검·분류·최소수정**이다. 대규모 의존성 업그레이드·아키텍처 변경은 범위 밖(별도 작업으로 제안).
- 한글 보고·주석 등 `CLAUDE.md` 컨벤션을 지킨다.
