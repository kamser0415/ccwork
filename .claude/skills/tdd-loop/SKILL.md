# tdd-loop

하나의 GitHub 이슈를 **끝까지(계약 확정 → 테스트 → 구현 → 검증 → 정리 → 점검 → PR)** 밀어주는 얇은
컨테이너다. **새로운 일을 하지 않는다.** 7단계 각각은 이미 있는 전용 스킬/에이전트가 담당하며,
이 스킬은 그것들을 **정해진 순서로 호출**하고 사이사이에 **진행 배너**를 출력할 뿐이다.

**왜 컨테이너가 따로 필요한가?** 각 단계 스킬은 독립적으로도 잘 돌지만, 이슈 하나를 처음부터 끝까지
가려면 사람이 매번 다음 스킬 이름을 기억해 입력해야 한다(순서·인자·베이스 브랜치 관례까지). 이 컨테이너는
그 "순서와 배선"만 책임진다. **판단과 승인은 여전히 사람 몫이다** — 아래 §핵심 규칙 참고.

## 입력

- `$ARGUMENTS` = **GitHub 이슈 번호** 하나. 예: `/tdd-loop 3`.
  선행 정수만 이슈 번호로 파싱한다(`"3 가보자"` → `3`). 그 이슈를 7단계 전부에 그대로 넘긴다.
- 이슈는 origin **`kamser0415/ccwork`** 기준이다. 조회할 땐 `gh ... --repo kamser0415/ccwork`로 명시한다
  (gh 기본 repo가 upstream으로 드리프트하면 엉뚱한 옛 이슈를 읽는다).

## 산출물

- 각 단계 스킬이 만드는 산출물(시나리오 문서, 테스트, 구현, 리팩토링, 점검 결과)이 순서대로 쌓이고,
  마지막에 **base가 `feature/<spec>`이고 본문에 `Closes #N`을 담은 PR**이 열린다.
- 컨테이너 자신은 **단계 배너**와, 중단 시 **"어디서·왜 멈췄는지"** 요약만 출력한다.

## 이 컨테이너의 핵심 규칙 (반드시 지킬 것)

이 스킬은 CLAUDE.md의 "자동으로 다음 단계로 넘어가지 않는다" 흐름 제어와 **모순되지 않는다.** 그 규칙이
막으려는 건 *사람 확인 없는 무단 진행*인데, 이 파이프라인의 사람 확인 지점은 **하위 스킬 내부 게이트**에
이미 촘촘히 박혀 있기 때문이다. 컨테이너는 그 게이트를 없애지 않고 **그대로 통과시킨다.**

1. **순서만 보장한다.** 0→1→2→…→7을 정해진 순서로 호출한다. 순서를 바꾸거나 단계를 건너뛰지 않는다.
2. **컨테이너는 추가 게이트를 만들지 않는다.** 단계와 단계 *사이*에서 "다음 단계 진행할까요?"라고
   되묻지 않는다. 한 단계가 정상 종료하면 곧바로 다음 단계 배너를 찍고 그 스킬을 호출한다.
3. **하위 스킬의 내부 승인 게이트는 100% 존중한다.** 그 게이트들이 사람 개입 지점이다 —
   컨테이너가 대신 승인하거나 자동으로 넘기지 **않는다.** 각 스킬이 "승인해 주세요"에서 멈추면
   **사람이 응답할 때까지 파이프라인도 멈춘다.** 자연 정지점은 아래와 같다:
   - **1단계** test-scenarios: ① 시그니처 승인 ② 시나리오 승인 (2회)
   - **5단계** tdd-refactor: 리팩토링 **대상 목록** 승인 (1회)
   - **6단계** security-review: **"즉시 수정 필요"** 처리 전 승인 (1회)
   - **7단계** create-pr: **PR 초안·base 브랜치** 승인 + **E2E 게이트**
4. **단계가 실패하면 즉시 중단한다.** 다음 단계로 넘어가지 말고, **몇 단계에서·무엇 때문에** 멈췄는지와
   **다음 행동 제안**을 출력한다(§실패·중단 처리). 실패를 삼키고 진행하지 않는다.
5. **4단계(AC 검증)에서 갭이 나오면 자동 진행 금지.** 보강 테스트를 임의로 넣지 말고, 갭을 보고한 뒤
   **"2단계(tdd-red)로 되돌아갈지"를 사람과 상의**한다(CLAUDE.md 흐름 규칙 4와 동일). 사람이 정할 문제다.

## 7단계 파이프라인 맵

| #   | 단계          | 호출                      | 내부 사람 게이트          | 실패 시                    |
| --- | ------------- | ------------------------- | ------------------------- | -------------------------- |
| 0   | 사전 점검     | 컨테이너 직접(gh/git)     | (브랜치 재실행 시 확인)   | 알리고 중단                |
| 1   | 계약+시나리오 | Skill `test-scenarios` N  | 시그니처·시나리오 승인(2) | 중단                       |
| 2   | RED           | Skill `tdd-red` N         | —                         | 진짜 RED 불가 시 중단      |
| 3   | GREEN         | Skill `tdd-green` N       | —(5회 루프 초과 시 보고)  | 5회로도 미통과면 중단      |
| 4   | AC 검증       | Agent `ac-verifier` N     | —(갭이면 상의)            | 갭 → 보고 후 상의(§규칙 5) |
| 5   | REFACTOR      | Skill `tdd-refactor` N    | 대상 목록 승인(1)         | 롤백 불가/막힘 시 중단     |
| 6   | 커밋 전 점검  | Skill `security-review` N | "즉시 수정" 승인(1)       | 클린 못 만들면 중단        |
| 7   | 통합(PR)      | Skill `create-pr` N       | PR 초안·base 승인 + E2E   | E2E 실패 → PR 미생성·중단  |

> 하위 스킬을 부르는 법: `/`로 시작하는 단계는 **Skill 도구**로 해당 스킬을 그 이슈 번호를 인자로 호출한다
> (예: Skill `tdd-green`, args `N`). 4단계 `@ac-verifier`만 **Agent 도구**로 `subagent_type: ac-verifier`,
> 프롬프트에 이슈 번호를 넣어 호출한다.

## 실행 절차

### 0. 사전 점검 (컨테이너가 직접 수행)

파이프라인을 안전하게 시작할 수 있는 상태인지 먼저 확인한다. **하나라도 실패하면 이후 단계를 시작하지 말고
사유를 출력하고 중단**한다.

1. **이슈 확인** — `gh issue view N --repo kamser0415/ccwork`로 본문과 **AC(수용 기준)**를 읽는다.
   이슈가 없거나 조회 실패면 중단.
2. **작업 트리 청결** — `git status --porcelain`. **미커밋 변경이 있으면 중단**한다(안전 차원 —
   파이프라인이 만들 변경과 기존 변경이 섞이면 롤백·리뷰가 오염된다). 사용자에게 커밋/스태시를 안내한다.
3. **베이스 브랜치 확인** — 현재 브랜치가 통합 기준인 **`feature/<spec>`** 인지 확인한다
   (`git branch --show-current`). 여기서 `<spec>`를 뽑아 5·7단계의 base로 쓴다.
   `feature/<spec>`가 아니면 어떤 브랜치에서 분기할지 애매하므로 **중단**하고 사용자에게 알린다.
4. **작업 브랜치 처리** — 이슈용 작업 브랜치 이름은 **`feat/<issue-slug>`** 다
   (이슈 번호+제목에서 짧은 kebab 슬러그, 예: `feat/3-note-tag-display`).
   - 이미 있으면 → **동일 이슈 재실행**으로 보고, 사용자에게 **덮어쓸지(체크아웃해 이어감)/취소할지** 묻는다.
     (이건 파괴적일 수 있는 결정이라 컨테이너가 임의로 정하지 않는다.)
   - 없으면 → `feature/<spec>`에서 `feat/<issue-slug>`로 **분기 + 체크아웃**한다.
5. 여기까지 통과하면 배너 `[0/7] 사전 점검 통과 — feat/<issue-slug> on feature/<spec>`를 찍고 1단계로.

### 1. 계약 확정 + 시나리오 — Skill `test-scenarios` N

배너 `━━━ [1/7] 계약+시나리오 — /test-scenarios N ━━━` 출력 후 **test-scenarios** 스킬을 이슈 번호로 호출한다.
이 스킬 안에는 **시그니처 승인**과 **시나리오 승인** 두 게이트가 있다 — 그 게이트에서 스킬이 멈추면
컨테이너도 멈추고 **사람의 승인/수정**을 기다린다. 산출물은 `docs/features/tag/issue-{N}.md`.
스킬이 승인 없이 진행 불가로 종료하거나 사람이 취소하면 → 중단.

### 2. RED — Skill `tdd-red` N

배너 `━━━ [2/7] RED — /tdd-red N ━━━` 후 **tdd-red** 호출. 승인된 시나리오를 **실행되며 실패하는** 테스트로
옮긴다(대상 파일이 없을 때만 `throw` 골격 생성). **모든 테스트가 "실행되어 올바른 이유로" 빨간지** 확인되면
정상. import 오류 등 *false RED*만 남아 진짜 RED를 만들 수 없으면 → **원인과 함께 중단**(시나리오/시그니처
모순일 수 있다 — 1단계로 되돌아갈지 상의).

### 3. GREEN — Skill `tdd-green` N

배너 `━━━ [3/7] GREEN — /tdd-green N ━━━` 후 **tdd-green** 호출. 빨간 테스트를 하나씩 통과시키고 매번 전체
회귀를 확인한다(테스트 1개당 **최대 5회** 피드백 루프). **전부 GREEN**이면 정상 — 스킬이 커버리지로 빠진
케이스 후보를 보고한다. **같은 테스트를 5회로도 못 통과**시켜 스킬이 보고하며 멈추면 → 중단하고 그 보고
(실패 메시지·시도·추정 원인)를 사람에게 넘긴다. 임의로 테스트를 고쳐 통과시키지 않는다.

### 4. AC 검증 — Agent `ac-verifier` N

배너 `━━━ [4/7] AC 검증 — @ac-verifier N ━━━` 후 **ac-verifier** 에이전트를 `subagent_type: ac-verifier`로
호출하고 프롬프트에 이슈 번호를 넣는다(읽기 전용 — 코드·테스트를 고치지 않는다). "테스트 통과 ≠ AC 충족"을
독립 판정한다.

- **충족**이면 배너와 함께 5단계로 진행.
- **갭이 있으면** → **여기서 자동 진행을 멈춘다.** 갭 내용을 사람에게 보고하고,
  **2단계(tdd-red)로 돌아가 테스트를 보강할지**를 상의한다(§핵심 규칙 5). 보강 테스트를 임의로 넣지 않는다.

### 5. REFACTOR — Skill `tdd-refactor` N

배너 `━━━ [5/7] REFACTOR — /tdd-refactor N ━━━` 후 **tdd-refactor** 호출. 동작은 고정한 채 구조만 개선한다.
이 스킬엔 **리팩토링 대상 목록 승인 게이트**가 있다 — 거기서 멈추면 사람 승인을 기다린다. 변경 한 건마다
`npm test`로 초록을 확인하고 깨지면 스킬이 즉시 롤백한다. (롤백 기준점이 되려면 GREEN 상태가 커밋되어
있어야 하므로, 이 스킬이 시작 전 커밋을 요구하면 그 안내를 따른다.) 스킬이 진행 불가로 멈추면 → 중단.

### 6. 커밋 전 점검 — Skill `security-review` N

배너 `━━━ [6/7] 커밋 전 점검 — /security-review N ━━━` 후 **security-review** 호출. `tsc --noEmit`·`npm audit`·`.env`
노출을 모아 **"즉시 수정 필요 / 권장 수정 / 무시 가능"** 3분류로 보고한다. **"즉시 수정 필요" 처리 전 승인 게이트**가
있다 — 거기서 멈추면 사람 승인을 기다린다. 승인된 🔴만 고치고 재스캔으로 클린을 확인한다. 🔴을 해소해 클린을
만들지 못하면 → 중단(빌드/보안이 깨진 채 PR로 넘기지 않는다).

### 7. 통합(PR) — Skill `create-pr` N

배너 `━━━ [7/7] 통합(PR) — /create-pr N ━━━` 후 **create-pr** 호출. 이 스킬이 커밋 요약으로 PR 초안을 만들고
(**PR 제목은 commitlint/Conventional Commits 형식**), **PR 초안·base 브랜치 승인 게이트**에서 멈춘다.
승인 뒤 **`npm run test:e2e`를 마지막 게이트**로 세워 통과할 때만 push + `gh pr create` 한다.

- **base = `feature/<spec>`** (0단계에서 뽑은 값), **PR 본문에 `Closes #N` 포함**, 이슈에 PR 링크 코멘트.
- **E2E가 깨지면** create-pr이 **PR을 만들지 않고** 복구 절차를 안내한다 → 컨테이너도 **중단**하고 그 사실을
  전달한다(E2E 코드를 고쳐 통과시키는 회피는 하지 않는다).
- ⚠️ base가 `feature/<spec>`(비-기본 브랜치)면 GitHub 자동 클로즈가 안 걸린다 — 머지 후
  `gh issue close N` 수동 종료가 필요함을 안내한다(create-pr이 이미 안내하면 반복하지 않는다).

PR이 열리면 배너 `[7/7] 완료 — PR: <url>`와 함께 **파이프라인 종료**를 알린다.

## 진행 메시지 형식

각 단계 호출 **직전에** 한 줄 배너만 출력한다. 예:

```
━━━ [3/7] GREEN — /tdd-green 3 ━━━
```

컨테이너의 발화는 **배너 + (중단 시) 사유 요약**으로 최소화한다. 각 단계의 상세 대화(승인 요청, 결과 보고)는
**하위 스킬/에이전트가 스스로** 사람과 주고받는다 — 컨테이너가 그것을 가로채거나 대신 요약하지 않는다.

## 실패·중단 처리 (모든 단계 공통)

한 단계라도 실패/취소/게이트 거부로 끝나면 **다음 단계를 시작하지 말고** 아래 형식으로 출력하고 멈춘다:

```
⛔ tdd-loop 중단 — [N/7] <단계명>
- 원인: <무엇이 왜 실패/거부됐는지 한두 줄>
- 현재 상태: 브랜치 feat/<issue-slug>, 마지막 성공 단계 [M/7]
- 다음 행동: <재개 방법 또는 사람이 정할 결정>
```

특히 다음은 **중단이 정상 동작**이다(버그가 아니다):

- **0단계** 미커밋 변경/잘못된 브랜치/이슈 조회 실패 → 시작 자체를 막는다.
- **2단계** 진짜 RED 불가(시나리오·시그니처 모순 의심) → 1단계 재고 상의.
- **3단계** 5회 루프로도 미통과 → 테스트/구현 모순을 사람에게.
- **4단계** AC 갭 → 2단계 복귀 여부 상의(임의 보강 금지).
- **6단계** 🔴을 클린으로 못 만듦 → 커밋/PR 금지.
- **7단계** E2E 실패 → PR 미생성, 통합 seam 원인을 프로덕션 코드에서 고치도록 안내.

## 하지 않는 것 (경계)

- **새 로직·테스트·리팩토링을 컨테이너가 직접 하지 않는다.** 전부 하위 스킬에 위임한다.
- **하위 스킬의 게이트를 대신 승인하지 않는다.** 사람 응답을 기다린다.
- **단계 사이에 자체 게이트를 추가하지 않는다.** ("다음 진행할까요?" 금지 — 자연 정지점은 하위 게이트뿐.)
- **순서를 바꾸거나 단계를 건너뛰지 않는다.** 사용자가 명시적으로 건너뛰길 원하면 그 위험을 먼저 짚고
  (예: RED 없이 GREEN은 false GREEN 위험) 사람이 정하게 한다.
